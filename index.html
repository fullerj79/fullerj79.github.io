Enhancement 1 · Software Design & Engineering

This artifact began as a console-based Python adventure game created for an introductory scripting course. The original implementation demonstrated control flow, dictionaries, loops, and basic function decomposition. It met its academic objective, but architecturally it was intentionally constrained.

In the original version, navigation, validation, state mutation, and output rendering were tightly coupled inside a single file. Data structures were hard-coded directly into the script. There was no persistence layer, no authentication, no environment separation, no structured logging, no automated verification, and no deployment strategy.

Transformation at a glance: a single-file console script became a layered web application with separation of concerns, authentication, secure configuration, structured logging, automated testing, and cloud deployment.

Architectural Refactor

The most significant improvement was architectural separation. The original artifact centralized responsibility in a procedural loop with direct access to shared state. In the enhanced version, responsibilities are distributed across clear layers.

Controllers orchestrate workflows. Domain models encapsulate state transitions and business rules. Behavior modules isolate policy logic. Repository classes abstract persistence. Views handle presentation. Utilities provide cross-cutting concerns such as structured logging.

Enhancement 2 · Algorithms & Data Structures

The original artifact solved gameplay with procedural control flow and hard-coded logic. It worked, but it did not formally model state, difficulty, or algorithmic guarantees.

In this enhancement, the system was rebuilt around explicit domain models, graph-based reasoning, and deterministic scoring logic. Game state is authoritative, difficulty selects behavior through strategy objects, and Breadth-First Search (BFS) computes optimal moves to anchor scoring in measurable correctness.

Transformation at a glance: gameplay moved from procedural mutation to explicit state modeling, graph traversal guarantees, and policy-driven difficulty selection.

GameState as Authoritative Model

GameState now tracks player position, visited rooms, collected items (as sets), move count, lifecycle status, timestamps, and event logs. State transitions are explicit and testable. 

        GameState
        - visited_rooms: set[str]
        - collected_items: set[str]
        - move_count: int
        - status: GameStatus
        - encountered_villain: bool

Difficulty via Enum + Strategy

Difficulty resolves visibility policies and scoring strategies rather than branching conditionals throughout the codebase. 

        Difficulty
        - visibility_policy
        - scoring_policy
        - scoring_strategy

Graph Modeling + BFS

The map is treated as a graph. BFS computes shortest-path baselines, guaranteeing solvability and enabling objective efficiency scoring.

Deterministic Scoring

Scoring combines progress, efficiency, and difficulty multipliers. Efficiency is measured against BFS-derived optimal_moves, making performance measurable and defensible.

Enhancement 3 · Databases

This enhancement moved the project from in-memory, hard-coded runtime state to a persistent, production-aligned application backed by MongoDB Atlas. The original artifact stored all game state inside a single loop and lost everything on exit. There was no user identity, saved progress, or leaderboard history.

In the enhanced version, persistence is a first-class capability. Users can create accounts, authenticate securely, save progress, resume sessions, and compete on leaderboards. This required intentional schema design, index strategy, repository abstractions, environment isolation, and security controls aligned with real-world engineering expectations.

Transformation at a glance: ephemeral state became secure, durable, queryable data through MongoDB Atlas, enforced by least-privilege access, IP restrictions, indexed collections, and repository-driven persistence boundaries.

Persistence Model and Data Boundaries

Persistence was designed around natural document boundaries: user identity, saved game state, and completed run history. MongoDB is a strong fit because these domains evolve independently while maintaining clear structural boundaries.

MongoDB Atlas Configuration and Security Controls

MongoDB Atlas mirrors production posture. Database users are configured with least privilege. Network access is restricted via IP allow lists. TLS is enforced for all connections. Secrets are injected at runtime and never committed to source control.

Repository Layer and Query Design

Persistence is isolated behind repository classes. Controllers orchestrate behavior. Repositories encapsulate data access. Domain models remain storage-agnostic. 

        Repositories
        - UserRepo: create and fetch users
        - SaveRepo: persist and resume game state
        - HistoryRepo: record runs and query leaderboard
        - LevelRepo: provide level definitions

The repository abstraction allows a MongoDB-backed implementation for production and an in-memory substitute for testing. This keeps tests fast and deterministic without sacrificing correctness.

User Identity and Password Storage

Passwords are never stored in plain text. Each credential is salted and hashed, combined with a server-side pepper stored securely in environment configuration. Identity fields enforce uniqueness at the database level.
